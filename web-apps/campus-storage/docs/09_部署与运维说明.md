## 一、文档概述

### 1.1 部署架构

```
本地开发 → pytest测试 → Flask本地联调 → 微信开发者工具预览
                                                ↓
              CloudBase环境创建 → 数据迁移 → 云函数部署 → 小程序发布
```

### 1.2 环境规划

Demo阶段采用"本地Python完整开发 → 统一部署"模式：

- **本地开发环境**：Python 3.12 + Flask + SQLite，用于完整业务逻辑开发与测试
- **生产部署环境**：CloudBase云开发 + 微信小程序，用于线上发布与运营

本地开发完成后，通过数据迁移脚本将SQLite数据导入CloudBase NoSQL，再将Flask路由适配为CloudBase云函数，统一部署上线。

## 三、部署步骤

### 3.1 本地开发环境搭建

```bash
# 1. 创建项目目录
mkdir campus-storage && cd campus-storage

# 2. 创建虚拟环境
python3 -m venv venv
source venv/bin/activate        # Linux/macOS
# venv\Scripts\activate         # Windows

# 3. 安装依赖
pip install flask pytest python-dotenv requests

# 4. 导出依赖清单
pip freeze > requirements.txt
```

### 3.2 本地数据库初始化

```bash
# 执行数据库初始化脚本（建表 + 种子数据）
python scripts/init_db.py
```

**初始化内容**：

| 数据表 | 操作 |
|--------|------|
| users | 创建表 + 导入1条管理员记录 |
| orders | 创建表 |
| warehouses | 创建表 + 导入3-5条预置仓库数据 |
| tickets | 创建表 |

**索引创建**：

| 数据表 | 索引字段 | 类型 |
|--------|---------|------|
| orders | userId, status | 复合索引 |
| orders | status, createTime | 复合索引 |
| orders | orderNo | 唯一索引 |

### 3.3 本地API服务器启动

```bash
# 启动Flask开发服务器
python app.py
# 或指定端口
FLASK_PORT=5000 python app.py
```

- 默认地址：`http://localhost:5000`
- API文档：`http://localhost:5000/api/docs`（如已集成）
- 使用pytest运行测试：`pytest tests/ -v`

### 3.4 小程序本地联调

1. 微信开发者工具打开小程序项目
2. 修改小程序API请求地址指向本地Flask服务器：

```javascript
// config.js
const API_BASE_URL = 'http://localhost:5000/api';
```

3. 微信开发者工具开启"不校验合法域名"（详情 → 本地设置）
4. 验证所有API接口在本地环境正常工作
5. 运行pytest确保测试全部通过

### 3.5 生产环境CloudBase创建

1. 登录微信开发者工具
2. 创建云开发环境（选择免费基础版）
3. 记录环境ID（用于云函数配置）
4. 开通CloudBase服务（微信云开发）

### 3.6 数据迁移（SQLite → CloudBase NoSQL）

详见第四节"数据迁移方案"。

简要步骤：

```bash
# 执行数据迁移脚本
python scripts/migrate_to_cloudbase.py --env-id <环境ID>
```

### 3.7 云函数部署（Flask路由 → CloudBase云函数适配）

**部署顺序**：

1. login（基础依赖）
2. createOrder
3. updateOrderStatus（核心）
4. getOrderList / getOrderDetail
5. createTicket
6. pendingTimeoutChecker / deliveringTimeoutChecker（定时触发）

**部署方式**：

- 微信开发者工具 → 云函数目录 → 右键"上传并部署：云端安装依赖"
- 或使用CloudBase CLI：`tcb fn deploy <functionName>`

**Python依赖管理**：

每个云函数目录下需包含 `requirements.txt` 文件，列出该函数所需的Python依赖包：

```
# 示例 requirements.txt
cloudbase-sdk==2.0.0
requests>=2.28.0
```

云端部署时，CloudBase会自动执行 `pip install -r requirements.txt` 安装依赖。

**Python运行时配置**：

在云函数配置中指定运行时为 Python 3.12：

```json
{
  "runtime": "Python3.12",
  "timeout": 20,
  "memorySize": 256
}
```

**环境变量配置**：

| 变量名 | 值 | 说明 |
|--------|---|------|
| ADMIN_OPENIDS | "实际管理员openid" | 管理员白名单 |
| STORAGE_FEE_RATE | "0.5" | 暂存费单价 |
| TIMEOUT_DAYS | "7" | 超时天数 |

### 3.8 定时触发器配置

| 云函数 | Cron表达式 | 配置方式 |
|--------|-----------|---------|
| pendingTimeoutChecker | `0 */30 * * * * *` | CloudBase控制台 → 云函数 → 触发器 |
| deliveringTimeoutChecker | `0 */30 * * * * *` | 同上 |

### 3.9 小程序上传发布

1. 修改小程序API请求地址指向CloudBase云函数：

```javascript
// config.js - 生产环境
const API_BASE_URL = 'https://<环境ID>.service.tcloudbase.com';
```

2. 微信开发者工具 → 点击"上传" → 填写版本号和备注
3. 登录微信公众平台 → 版本管理 → 提交审核
4. 审核通过后 → 点击"发布"

## 四、数据迁移方案

### 4.1 SQLite → CloudBase NoSQL 数据映射

| SQLite表 | CloudBase集合 | 映射说明 |
|----------|--------------|---------|
| users | users | 字段一一对应，自增id转为字符串 |
| orders | orders | 字段一一对应，自增id转为字符串，日期格式保持ISO 8601 |
| warehouses | warehouses | 字段一一对应，自增id转为字符串 |
| tickets | tickets | 字段一一对应，自增id转为字符串 |

**类型转换规则**：

| SQLite类型 | CloudBase类型 | 转换方式 |
|-----------|--------------|---------|
| INTEGER | Number | 直接映射 |
| TEXT | String | 直接映射 |
| REAL | Number | 直接映射 |
| BLOB | String (Base64) | Base64编码存储 |
| DATETIME (TEXT) | String | 保持ISO 8601格式 |
| 自增主键 (id) | String | 转为字符串作为文档_id |

### 4.2 迁移脚本设计

迁移脚本 `scripts/migrate_to_cloudbase.py` 核心流程：

```python
"""
数据迁移脚本：SQLite → CloudBase NoSQL

使用方式：
    python scripts/migrate_to_cloudbase.py --env-id <环境ID>
    python scripts/migrate_to_cloudbase.py --env-id <环境ID> --dry-run  # 仅预览
"""

# 核心步骤：
# 1. 读取本地SQLite数据库
# 2. 按依赖顺序遍历表（warehouses → users → orders → tickets）
# 3. 逐行转换数据格式（类型转换、字段映射）
# 4. 批量写入CloudBase对应集合
# 5. 输出迁移报告（成功数、失败数、跳过数）
```

**迁移安全措施**：

- 支持 `--dry-run` 模式，仅预览不实际写入
- 迁移前自动备份CloudBase目标集合数据
- 每批写入后进行计数校验
- 迁移失败自动回滚已写入的批次

### 4.3 数据验证方法

迁移完成后执行以下验证：

```bash
# 运行数据验证脚本
python scripts/verify_migration.py --env-id <环境ID>
```

**验证检查项**：

| 检查项 | 验证方式 | 通过标准 |
|--------|---------|---------|
| 记录总数 | SQLite COUNT vs CloudBase COUNT | 完全一致 |
| 字段完整性 | 抽样检查关键字段非空 | 100%通过 |
| 数据类型 | 抽样检查字段类型匹配 | 100%通过 |
| 关联关系 | 检查外键引用完整性 | 无孤立记录 |
| 索引有效性 | CloudBase集合索引查询测试 | 查询正常返回 |

## 五、配置管理

### 5.1 本地环境配置（.env文件）

项目根目录创建 `.env` 文件，用于本地开发环境配置：

```env
# Flask配置
FLASK_PORT=5000
FLASK_DEBUG=true

# 数据库配置
DATABASE_PATH=./data/campus_storage.db

# 业务参数
STORAGE_FEE_RATE=0.5
TIMEOUT_DAYS=7
ADMIN_OPENIDS=admin_openid_1,admin_openid_2

# CloudBase配置（迁移时使用）
CLOUDBASE_ENV_ID=your-env-id
```

`.env` 文件应加入 `.gitignore`，避免提交到代码仓库。通过 `python-dotenv` 在Flask应用中加载：

```python
from dotenv import load_dotenv
load_dotenv()
```

### 5.2 环境变量清单

参见3.7节云函数环境变量配置。

### 5.3 敏感配置管理

- 管理员openid：本地存储在 `.env` 文件中，生产环境存储在云函数环境变量中，不硬编码在代码里
- 代码中通过 `os.environ.get('ADMIN_OPENIDS')` 读取环境变量
- Demo阶段无其他敏感配置

### 5.4 配置变更流程

**本地环境**：

1. 修改 `.env` 文件
2. 重启Flask服务器使配置生效
3. 验证配置生效

**生产环境**：

1. 修改云函数环境变量（CloudBase控制台）
2. 重新部署云函数（使环境变量生效）
3. 验证配置生效

## 六、运维监控

- 云函数日志：CloudBase控制台 → 云函数 → 日志

### 6.2 监控指标

### 6.3 Demo阶段监控方案

## 七、应急预案

### 7.1 常见故障处理

| 故障现象 | 可能原因 | 处理方式 |
|---------|---------|---------|
| 云函数超时 | 数据量过大/逻辑过重 | 优化查询、增加分页 |
| 数据库连接失败 | CloudBase服务异常 | 查看CloudBase状态页 |
| 订单状态不一致 | 并发冲突/定时任务干扰 | 检查statusHistory日志 |
| 小程序白屏 | 代码错误/网络异常 | 查看vConsole错误日志 |
| 本地API无响应 | Flask服务未启动/端口冲突 | 检查进程、确认端口 |

### 7.2 回滚流程

1. 微信公众平台 → 版本管理 → 回退到上一版本
2. 云函数回滚：重新上传旧版本代码

### 7.3 数据备份

- CloudBase自动备份（免费版不提供）
- Demo阶段：手动导出关键数据（orders集合）
- 本地SQLite数据库：定期复制 `data/campus_storage.db` 备份文件

### 8.2 版本发布流程

采用"本地开发完成 → 测试通过 → 统一部署"模式：

1. **本地开发完成**：在本地Python+Flask环境完成功能开发
2. **测试通过**：运行 `pytest tests/ -v`，全部用例通过
3. **本地联调验证**：小程序连接本地Flask API，功能验证通过
4. **数据迁移**：执行 `migrate_to_cloudbase.py` 将SQLite数据同步到CloudBase
5. **云函数部署**：将Flask路由适配为云函数并上传部署
6. **小程序预览**：微信开发者工具预览，验证生产环境功能
7. **上传发布**：上传小程序代码 → 提交审核 → 审核通过后发布

1. 定位问题（查看日志）
