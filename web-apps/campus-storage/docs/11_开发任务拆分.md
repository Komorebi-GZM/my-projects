# 校园物品暂存平台 · 开发任务拆分

> 版本：v3.0
> 创建日期：2026-05-10
> 更新日期：2026-05-10
> 技术栈：微信小程序 + CloudBase云开发（Python 3.12）
> 开发模式：本地Python完整开发（SQLite + Flask + pytest） → 统一部署微信小程序

---

## 一、概述

本文档将「校园物品暂存平台」项目拆分为可独立实现、可验证的开发任务。每个任务遵循垂直切片原则，交付完整功能。

**项目目标**：实现一个校园物品暂存服务，支持学生寄存物品、运营端管理、定时任务处理。

**v3.0 核心变更**：开发模式从"边开发边部署CloudBase"改为"本地Python完整开发 → 统一部署微信小程序"。新增 Phase 0（本地开发环境搭建）和 Phase 7（生产环境部署），所有云函数开发改为本地 Flask 路由开发，数据库操作改为 SQLite 本地操作，测试改为 pytest + SQLite 内存数据库。

---

## 二、架构决策

| 决策 | 选择 | 理由 |
|------|------|------|
| **本地开发** | SQLite + Flask + pytest | 轻量级本地调试，无需网络依赖，迭代速度快 |
| **运行时** | Python 3.12+ | CloudBase官方支持，async/await原生支持 |
| **状态管理** | 乐观锁 + statusHistory | 无事务支持，通过版本号实现并发控制 |
| **测试框架** | pytest + SQLite内存数据库 | 本地快速测试，无需CloudBase连接 |
| **状态机** | 矩阵校验 + 终态保护 | 13条合法转换，COMPLETED/CANCELLED不可变更 |
| **定时任务** | 本地APScheduler调试 → CloudBase定时触发器 | 本地用APScheduler模拟，生产用CloudBase定时触发器 |
| **前端** | 原生小程序 | 学习成本低，无框架依赖 |
| **数据库策略** | 本地SQLite轻量模拟，生产CloudBase NoSQL | 通过db_adapter.py抽象层实现双模式切换 |
| **部署策略** | 先本地完整开发和测试，最后统一部署 | 减少云资源消耗，加快开发迭代 |

## 三、依赖关系图

```
本地环境（Python 3.12 + venv + SQLite + Flask）
    │
    ├── 数据库层（SQLite本地 / CloudBase NoSQL生产）
    │       └── db_adapter.py（数据访问抽象层，双模式切换）
    │
    ├── 公共模块（common/）
    │       ├── errors.py（错误处理）      ← 无依赖，最先开发
    │       ├── auth.py（身份认证）        ← 依赖 errors.py
    │       ├── validator.py（参数校验）  ← 依赖 errors.py
    │       └── permission.py（权限校验） ← 依赖 auth.py
    │
    ├── Flask路由层（本地开发，替代云函数）
    │       ├── /api/login（依赖：auth.py）
    │       ├── /api/orders（依赖：errors, auth, validator）
    │       ├── /api/orders/<id>/status（依赖：common/*, 状态机）  ← 核心复杂度
    │       ├── /api/orders/list（依赖：auth.py）
    │       ├── /api/orders/<id>（依赖：auth.py）
    │       ├── /tasks/pending-timeout（依赖：updateOrderStatus）
    │       └── /tasks/delivering-timeout（依赖：updateOrderStatus）
    │
    ├── 云函数层（生产部署，由Flask路由适配而来）
    │       ├── login → CloudBase云函数
    │       ├── createOrder → CloudBase云函数
    │       ├── updateOrderStatus → CloudBase云函数
    │       ├── getOrderList → CloudBase云函数
    │       ├── getOrderDetail → CloudBase云函数
    │       ├── pendingTimeoutChecker → CloudBase定时触发器
    │       └── deliveringTimeoutChecker → CloudBase定时触发器
    │
    └── 前端小程序
            ├── 页面组件（依赖：Flask API / 云函数API）
            └── 状态管理（依赖：Flask API / 云函数API）
```

## 四、任务拆分策略

- **垂直切片**：每个任务交付完整功能（数据库 + 后端逻辑 + 测试）
- **任务大小**：S（1-2文件）或 M（3-5文件），不超过5文件
- **开发顺序**：本地环境 → 基础设施 → 核心功能 → 辅助功能 → 定时任务 → 前端 → 部署
- **高风险前置**：状态机逻辑（Task 15）在Phase 2早期验证
- **本地优先**：所有后端逻辑先在本地Flask + SQLite环境完成开发和测试
