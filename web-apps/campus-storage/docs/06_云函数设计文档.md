## 一、文档概述

### 1.1 云函数架构

校园物品暂存平台基于微信小程序 + CloudBase 云开发架构，后端所有业务逻辑通过云函数实现。云函数采用 **HTTP 触发方式**，每个云函数独立部署、独立调用，通过云函数名区分功能边界。CloudBase 支持 Python 运行时，当前选择 **Python 3.12**，利用其丰富的生态和团队技术栈统一性。

**为什么选择云函数而非传统后端服务？**

- **零运维**：CloudBase 提供开箱即用的运行环境，无需自行搭建服务器、配置 Nginx、管理 SSL 证书
- **天然鉴权**：CloudBase HTTP 触发器自动注入 `event.userInfo`，包含微信用户的 `openId`，免去手动实现 JWT 签发与校验
- **与微信生态深度集成**：云函数可直接调用微信支付 SDK、订阅消息推送等能力，无需额外中转
- **Demo 阶段成本最优**：按调用次数计费，开发调试期间几乎零成本，适合面试展示项目

> **已知限制**：CloudBase Python 冷启动时间可能比 Node.js 略长（约300~800ms），但 Demo 场景可接受。

**开发模式：本地 Python 开发 → 统一部署**

本项目采用"本地开发 + 生产部署"双模式架构，业务逻辑在本地完成开发和调试后，统一适配部署为 CloudBase 云函数：

- **开发阶段**：使用本地 Python + Flask + SQLite 进行开发。Flask 提供轻量级 HTTP 服务器，SQLite 提供零配置的本地数据库，开发者可以在不依赖云服务的情况下完成全部业务逻辑的开发和测试。
- **部署阶段**：通过数据访问层抽象（`db_adapter.py`）将数据库操作从 SQLite 切换为 CloudBase 数据库，每个 Flask 路由对应一个云函数入口，实现无缝部署。
- **数据访问层抽象**：`db_adapter.py` 封装了所有数据库操作，提供统一的接口。通过环境变量 `DB_MODE`（`local` / `production`）自动切换底层实现，业务逻辑代码无需任何修改即可在两种模式间切换。

### 1.2 设计原则

| 原则 | 说明 | 为什么重要 |
|------|------|-----------|
| **单一职责** | 每个云函数只做一件事 | 降低单个函数的复杂度，便于独立测试和部署；面试时能清晰阐述每个函数的边界 |
| **无状态** | 不依赖云函数实例的本地状态 | CloudBase 云函数实例可能被随时回收，本地变量/缓存不可靠；所有持久化数据必须读写数据库 |
| **幂等性** | 相同输入多次调用结果一致 | 网络抖动导致前端重复提交时，不会产生脏数据（如重复写入 statusHistory） |
| **快速失败** | 参数校验在前，业务逻辑在后 | 无效请求在入口处即被拦截，避免浪费数据库查询资源；错误信息精确到字段级别，便于前端提示 |
| **环境无关** | 业务逻辑不依赖特定运行环境 | 通过 `db_adapter.py` 抽象数据访问层，同一份业务代码可在本地 Flask + SQLite 和 CloudBase 云函数两种环境中运行，无需修改 |

### 1.3 公共模块

所有云函数共享以下公共逻辑，以 Python 工程化结构组织：

```
common/
├── __init__.py
├── auth.py          # get_user_info(event) → {openId, role}
├── permission.py    # check_permission(required_role, current_user, order)
├── validator.py     # 参数校验、状态枚举校验
├── errors.py        # 统一错误码类 ErrorCode + 输出格式
└── db_adapter.py    # 数据访问层抽象（SQLite/CloudBase双模式）
```

| 模块 | 职责 | 说明 |
|------|------|------|
| **身份认证模块** (`auth.py`) | 从 `event.userInfo.openId` 获取用户身份 | CloudBase HTTP 触发器自动注入 `userInfo`，无需手动解析 token |
| **权限校验模块** (`permission.py`) | 校验 `operatorType` 是否匹配操作要求 | 区分 USER / ADMIN / SYSTEM 三种角色，不同操作有不同权限要求 |
| **参数校验模块** (`validator.py`) | 参数校验、状态枚举校验 | 统一校验逻辑，避免各函数重复实现 |
| **错误处理模块** (`errors.py`) | 统一错误码类 `ErrorCode` + 输出格式 | 所有云函数共享同一套错误码体系，前端可统一拦截处理 |
| **数据访问层** (`db_adapter.py`) | 数据库操作抽象，屏蔽 SQLite 与 CloudBase 差异 | 通过环境变量 `DB_MODE` 自动切换底层实现，业务代码统一调用 `db_adapter` 接口 |

**为什么抽取公共模块而非在每个函数中重复实现？**

- 面试时体现工程化思维：DRY 原则（Don't Repeat Yourself）
- 错误码统一管理，避免不同函数对同一错误场景返回不同码值
- 权限逻辑集中维护，新增权限规则时只需修改一处

## 三、核心云函数详细设计：updateOrderStatus

### 3.1 函数签名

```
def main(event, context):
    """
    输入参数 event:
    {
        'orderId': str,        # 订单ID（必填）
        'newStatus': str,      # 目标状态枚举值（必填）
        'operatorType': str,   # 操作人类型：USER / ADMIN / SYSTEM（必填）
        'operatorId': str,     # 操作人ID（必填）
        'reason': str,         # 变更原因枚举值（必填）
        'metadata': dict       # 附加信息（选填，如物流单号、仓库位置等）
    }

    成功输出：
    {
        'code': 0,
        'message': 'success',
        'data': {
            'order': Order,              # 更新后的完整订单对象
            'statusRecord': StatusRecord # 本次写入的状态变更记录
        }
    }

    失败输出：
    {
        'code': int,    # 错误码（见 3.6 节）
        'message': str  # 人类可读的错误描述
    }
    """
```

**设计决策：为什么 reason 是必填字段？**

> 引用自《订单状态机设计文档.md》v2.0，决策 Q10

- 每一次状态变更都必须有明确的原因，便于后续追溯和审计
- 枚举约束（如 `USER_CANCEL`、`TIMEOUT_CANCEL`、`ADMIN_DISPATCH`、`DELIVERY_FAILED`、`ITEM_DAMAGE`、`ADMIN_MARK_EXCEPTION` 等）避免自由文本导致的数据不一致
- 面试时可以强调：这是"可观测性"设计的一部分，每条 statusHistory 记录都是完整的操作凭证

### 3.2 业务逻辑步骤（伪代码描述）

```
Step 1: 参数校验
  - 校验 orderId 非空且格式合法
  - 校验 newStatus 为合法枚举值（PENDING/COLLECTED/TRANSIT/STORED/DELIVERING/COMPLETED/CANCELLED/EXCEPTION）
  - 校验 operatorType 为 USER/ADMIN/SYSTEM 之一
  - 校验 reason 非空
  → 失败返回错误码 1001，附带具体字段错误信息

Step 2: 查询订单
  - 根据 orderId 从 orders 集合查询订单
  - 校验订单存在
  → 失败返回错误码 2001

Step 2.5: 终态保护检查
  ```python
  # Step 2.5: 终态保护检查
  if current_order['status'] in ('COMPLETED', 'CANCELLED'):
      return {'code': 2004, 'message': '订单已处于终态，不可进行任何状态变更'}
  ```

Step 3: 状态转换校验
  - 查询状态转换矩阵（见 3.3 节）
  - 校验 order.status → newStatus 是否为合法转换
  → 失败返回错误码 2002，附带当前状态和合法转换列表

Step 4: 权限校验
  - 根据状态转换矩阵，校验当前 operatorType 是否有权执行此转换
  - 任何状态 → EXCEPTION：必须 operatorType == 'ADMIN'
  - EXCEPTION → COMPLETED/CANCELLED：必须 operatorType == 'ADMIN'
  → 失败返回错误码 1003，附带所需权限说明

Step 5: 乐观锁更新（最多重试 3 次）
  for retry in range(3):
    - 读取订单当前 _version
    - 幂等性检查：statusHistory 末条记录的 toStatus 是否已经是目标状态
      → 如果是，直接返回成功（幂等性保障）
    - 使用 _version 作为条件执行更新（CloudBase 原生支持）
    - 更新成功则跳出循环
    - 更新失败（_version 不匹配）则重试
  → 3 次均失败返回错误码 2003

Step 6: 写入 statusHistory
  - 构造 statusRecord:
    {
        'timestamp': 当前时间,
        'fromStatus': order.status,  # 变更前状态
        'toStatus': new_status,
        'operatorType': operator_type,
        'operatorId': operator_id,
        'reason': reason,
        'metadata': metadata or {}
    }
  - 追加到 statusHistory 列表

Step 7: 返回结果
  - 返回更新后的完整订单信息和本次状态变更记录
```

### 3.3 状态转换矩阵（完整表格）

> 引用自《订单状态机设计文档.md》v2.0，第二节状态定义 + 第三节状态转换规则

以下矩阵定义了所有合法的状态转换、所需权限、必填原因枚举值以及额外校验规则：

| 当前状态 | 目标状态 | 所需 operatorType | 所需 reason 枚举值 | 额外校验规则 | 示例场景 |
|---------|---------|-------------------|-------------------|-------------|---------|
| `PENDING` | `COLLECTED` | `ADMIN` | `ADMIN_COLLECT` | 更新取件时间、取件人信息 | 管理员上门取件完成 |
| `PENDING` | `CANCELLED` | `USER` / `ADMIN` | `USER_CANCEL` / `TIMEOUT_CANCEL` / `ADMIN_CANCEL` | 已支付订单需触发自动退款 | 用户主动取消 / 超时7天系统自动取消 |
| `PENDING` | `EXCEPTION` | `ADMIN` | `ADMIN_MARK_EXCEPTION` | reason 必填，记录异常详情 | 物品损坏无法取件 |
| `COLLECTED` | `TRANSIT` | `ADMIN` | `ADMIN_DISPATCH` | 24h 未操作需系统提醒（由定时触发器实现） | 管理员发运至仓库 |
| `COLLECTED` | `EXCEPTION` | `ADMIN` | `ADMIN_MARK_EXCEPTION` | reason 必填 | 取件后发现问题 |
| `TRANSIT` | `STORED` | `ADMIN` | `ADMIN_STORE` | 需上传入库照片、记录仓库位置（metadata 中携带） | 物品入库上架 |
| `TRANSIT` | `EXCEPTION` | `ADMIN` | `ADMIN_MARK_EXCEPTION` | reason 必填 | 运输途中丢失 |
| `STORED` | `DELIVERING` | `USER` / `ADMIN` | `USER_REQUEST_DELIVERY` / `ADMIN_DISPATCH` | 用户发起时需填写配送地址 | 用户申请配送 / 管理员主动发货 |
| `STORED` | `EXCEPTION` | `ADMIN` | `ADMIN_MARK_EXCEPTION` | reason 必填 | 仓库发现物品异常 |
| `DELIVERING` | `COMPLETED` | `USER` / `ADMIN` / `SYSTEM` | `USER_CONFIRM` / `ADMIN_COMPLETE` / `TIMEOUT_COMPLETE` | SYSTEM 仅限定时触发器调用 | 用户确认收货 / 超时7天系统自动完结 |
| `DELIVERING` | `EXCEPTION` | `ADMIN` | `DELIVERY_FAILED` / `ADMIN_MARK_EXCEPTION` | 含配送失败退回场景 | 配送失败退回 |
| `EXCEPTION` | `COMPLETED` | `ADMIN` | `ADMIN_RESOLVE_COMPLETE` | 人工结案（如破损赔偿后完结） | 异常处理后结案 |
| `EXCEPTION` | `CANCELLED` | `ADMIN` | `ADMIN_RESOLVE_CANCEL` | 人工作废（如配送失败 → 取消 + 退款） | 异常处理后取消退款 |

**不合法转换（代码层面拒绝）：**

- 终态（`COMPLETED`、`CANCELLED`）不可转为任何其他状态
- 不允许跳过中间状态（如 `PENDING` 直接转 `STORED`）
- 不允许逆向转换（如 `STORED` 回退到 `COLLECTED`）

### 3.4 乐观锁重试机制

> 引用自《订单状态机设计文档.md》v2.0，决策 Q7

**为什么选择乐观锁而非悲观锁？**

| 对比维度 | 乐观锁 | 悲观锁 |
|---------|--------|--------|
| 实现方式 | `_version` 字段条件更新 | `SELECT FOR UPDATE` 行级锁 |
| CloudBase 支持 | 原生支持，`db.collection().where({_version: currentVersion}).update()` | 不支持传统数据库悲观锁 |
| 并发性能 | 无锁等待，冲突时重试 | 串行化执行，高并发下性能下降 |
| 适用场景 | 冲突概率低的场景 | 冲突概率高的场景 |
| Demo 阶段适配性 | 并发量极低，冲突几乎不会发生 | 过度设计，增加复杂度 |

**结论**：Demo 阶段并发量极低，乐观锁性能更优；CloudBase 不支持传统数据库悲观锁；乐观锁实现简单，`_version` 字段 CloudBase 原生支持。

**重试策略详细设计：**

```python
from common import db_adapter

MAX_RETRIES = 3

for retry in range(MAX_RETRIES):
    # 1. 读取订单最新 _version
    doc = db_adapter.get_order(order_id)

    # 2. 幂等性检查
    if doc['statusHistory'][-1]['toStatus'] == new_status:
        return success_response(doc)

    # 3. 执行条件更新（通过 db_adapter 统一接口，自动适配 SQLite/CloudBase）
    try:
        db_adapter.update_order(
            order_id=order_id,
            data={
                'status': new_status,
                'statusHistory': db_adapter.push(status_record)
            },
            condition={'version': doc['_version']}
        )
        break
    except db_adapter.VersionConflictError as e:  # 版本冲突异常
        if retry == MAX_RETRIES - 1:
            return error_response(2003, '系统繁忙，请稍后重试')
```

**幂等性保障：**

> 引用自《订单状态机设计文档.md》v2.0，第十一节编码注意事项第 1 条

- 重试前检查 `statusHistory` 末条记录的 `toStatus` 是否已经是目标状态
- 如果是，说明上一次请求已经成功执行，直接返回成功，避免重复写入
- 这保证了前端因网络抖动导致的重复提交不会产生脏数据

**ABA 问题评估：**

> 引用自《订单状态机设计文档.md》v2.0，决策 Q8

- **ABA 问题定义**：值从 A 变为 B 又变回 A，乐观锁因版本号递增仍能检测到变化，但业务层面可能误判
- **当前设计评估**：状态机中不存在逆向回流（如 `STORED` 不会回退到 `COLLECTED`），因此不存在 ABA 风险
- **结论**：已评估并记录，当前设计无需额外处理 ABA 问题

### 3.5 权限校验逻辑

> 引用自《订单状态机设计文档.md》v2.0，决策 Q15

**权限模型设计：**

操作人分为三种角色，每种角色能执行的状态转换不同：

| 角色 | 标识 | 说明 | 可执行的状态转换 |
|------|------|------|----------------|
| 普通用户 | `USER` | 微信小程序端用户 | `PENDING → CANCELLED`、`STORED → DELIVERING`、`DELIVERING → COMPLETED` |
| 管理员 | `ADMIN` | 运营后台管理人员 | 所有状态转换（含异常标记、异常闭环） |
| 系统 | `SYSTEM` | 定时触发器自动执行 | `DELIVERING → COMPLETED`（超时自动完结）、`PENDING → CANCELLED`（超时自动取消） |

**硬编码权限规则（在云函数中实现）：**

```
权限校验矩阵：
- 任何状态 → EXCEPTION：必须 operatorType == 'ADMIN'
- EXCEPTION → COMPLETED：必须 operatorType == 'ADMIN'
- EXCEPTION → CANCELLED：必须 operatorType == 'ADMIN'
- PENDING → COLLECTED：必须 operatorType == 'ADMIN'
- COLLECTED → TRANSIT：必须 operatorType == 'ADMIN'
- TRANSIT → STORED：必须 operatorType == 'ADMIN'
- PENDING → CANCELLED：operatorType == 'USER' 或 'ADMIN'
- STORED → DELIVERING：operatorType == 'USER' 或 'ADMIN'
- DELIVERING → COMPLETED：operatorType == 'USER' 或 'ADMIN' 或 'SYSTEM'
```

**为什么选择硬编码而非配置化？**

- Demo 阶段权限规则固定且数量有限（13 条转换规则），硬编码更直观
- 配置化（如数据库存储权限矩阵）增加查询开销和复杂度，在当前阶段属于过度设计
- 面试时可以说明：如果权限规则频繁变动，可以抽取为配置表，通过管理后台动态维护

**管理员身份验证方式：**

- 方案一（当前采用）：通过环境变量 `ADMIN_OPENIDS` 配置管理员 openId 列表，云函数启动时加载
- 方案二（备选）：在 users 集合中维护 role 字段，查询时校验
- 选择方案一的原因：Demo 阶段管理员数量固定（1-2 人），环境变量方式最简单，无需额外数据库查询

### 3.6 错误处理策略

| 错误场景 | 错误码 | HTTP 状态码 | 处理方式 | 返回信息示例 |
|---------|--------|------------|---------|-------------|
| 参数缺失/格式错误 | `1001` | 400 | 直接返回，不执行业务逻辑 | `"orderId 不能为空"` |
| 认证失败（openId 无效） | `1002` | 401 | 直接返回 | `"用户身份验证失败"` |
| 权限不足 | `1003` | 403 | 直接返回，附带所需权限说明 | `"该操作需要管理员权限"` |
| 订单不存在 | `2001` | 404 | 直接返回 | `"订单不存在：orderId=xxx"` |
| 非法状态转换 | `2002` | 409 | 返回当前状态 + 合法转换列表 | `"非法状态转换：PENDING 不可直接转为 STORED，合法目标：COLLECTED, CANCELLED, EXCEPTION"` |
| 乐观锁冲突（3 次重试均失败） | `2003` | 409 | 自动重试 3 次后返回 | `"系统繁忙，请稍后重试"` |
| 订单已终态，不可变更 | `2004` | 409 | 直接返回 | `"订单已处于终态，不可变更"` |
| 系统异常（数据库错误等） | `5001` | 500 | 记录完整错误日志，返回通用错误信息 | `"服务异常，请联系客服"` |

**错误码分段设计：**

- `1xxx`：客户端错误（参数、认证、权限）
- `2xxx`：业务逻辑错误（订单不存在、非法转换、并发冲突）
- `5xxx`：服务端内部错误

**为什么非法状态转换返回 409 而非 400？**

- 409 Conflict 表示请求与服务器当前状态冲突，语义上更准确
- 前端可以根据 409 状态码自动刷新订单状态，展示最新合法操作
- 面试时可以强调对 HTTP 语义的理解

Step 5: 返回订单信息

  - 校验 orderId、category、description 非空
  - 校验关联订单存在

Step 2: 校验订单状态
  - 仅 COMPLETED 状态的订单可创建投诉工单
  - 其他状态提示用户先完成订单流程

Step 3: 写入 tickets 集合
  - 初始状态：OPEN
  - 记录创建时间、关联订单信息

Step 4: 返回工单信息
```

**设计决策：为什么投诉走独立工单而非修改订单状态？**

> 引用自《订单状态机设计文档.md》v2.0，决策 Q5

- COMPLETED 是终态，不可逆。如果允许投诉修改订单状态，会破坏状态机的严谨性
- 投诉处理流程（受理 → 调查 → 赔偿 → 关闭）与订单状态流转是两个独立的业务维度
- 独立工单系统更灵活：一个订单可以有多个工单，工单有自己的生命周期

### 4.6 pendingTimeoutChecker — PENDING 超时自动取消

**触发方式：** 定时触发，每 30 分钟执行一次

**业务逻辑：**

```
Step 1: 查询超时订单
  - 查询 status === 'PENDING' 且 createdAt < 当前时间 - TIMEOUT_DAYS 天的订单
  - TIMEOUT_DAYS 通过环境变量配置，默认 7 天

Step 2: 逐单处理
  - 对每个超时订单调用 updateOrderStatus 的核心逻辑
  - operatorType = 'SYSTEM'
  - reason = 'TIMEOUT_CANCEL'

Step 3: 退款处理（如需要）
  - 如订单 isPaid == True，触发自动退款逻辑
  - 退款逻辑独立实现，不阻塞状态变更

Step 4: 记录日志
  - 日志标记 `source: 'CRON'`，便于区分定时触发与手动调用
```

### 4.7 deliveringTimeoutChecker — DELIVERING 超时自动完结

**触发方式：** 定时触发，每 30 分钟执行一次

**业务逻辑：**

```
Step 1: 查询超时订单
  - 查询 status === 'DELIVERING' 的订单
  - Demo 阶段直接使用 deliveringAt 字段判断超时，不遍历 statusHistory：
    ```python
    from datetime import datetime, timedelta
    timeout_threshold = datetime.utcnow() - timedelta(days=7)
    # 直接使用 deliveringAt 字段判断
    if order['deliveringAt'] and order['deliveringAt'] < timeout_threshold:
        # 触发自动完结
    ```

Step 2: 逐单处理
  - 对每个超时订单调用 updateOrderStatus 的核心逻辑
  - operatorType = 'SYSTEM'
  - reason = 'TIMEOUT_COMPLETE'

Step 3: 记录日志
  - 日志标记 `source: 'CRON'`，便于区分定时触发与手动调用
```

**设计决策：为什么 DELIVERING 超时计时起点是 statusHistory 中的 timestamp 而非 updatedAt？**

> 引用自《订单状态机设计文档.md》v2.0，决策 Q4

- `updatedAt` 在任何字段变更时都会更新，无法精确反映进入 DELIVERING 状态的时间
- `statusHistory` 中 DELIVERING 记录的 `timestamp` 是进入该状态的精确时间点
- 这保证了超时计算的准确性，不受其他字段更新影响

### 7.1 环境变量

| 变量名 | 说明 | 示例值 | 为什么用环境变量 |
|--------|------|--------|----------------|
| `ADMIN_OPENIDS` | 管理员 openId 列表（逗号分隔） | `"openid1,openid2"` | 管理员变更时无需修改代码，只需更新环境变量；读取方式：`os.environ.get('ADMIN_OPENIDS')` |

| `TIMEOUT_DAYS` | 超时天数（PENDING 和 DELIVERING 共用） | `"7"` | 不同运营策略下可灵活调整；读取方式：`os.environ.get('TIMEOUT_DAYS')` |

### 7.2 资源配置

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 内存 | 256MB（默认） | Demo 阶段业务逻辑简单，默认配置足够 |
| 超时时间 | 20 秒 | updateOrderStatus 包含 3 次重试，需预留足够时间 |
| 并发数 | 按 CloudBase 默认配置 | Demo 阶段无需手动调整 |

> **Python 运行时说明**：Python 云函数运行时内存建议 256MB，超时时间 20s。Python 解释器启动可能稍慢（冷启动约300~800ms），20s 足够覆盖 3 次乐观锁重试 + 业务逻辑。

**为什么超时时间设为 20 秒而非默认的 3 秒？**

- `updateOrderStatus` 在最坏情况下需要执行 3 次数据库查询 + 3 次条件更新
- 每次数据库操作约 50-200ms（CloudBase 内网延迟低），3 次重试总计约 1-2 秒
- 加上参数校验、权限校验、日志记录等开销，20 秒留有充足余量
- 过短的超时时间可能导致重试未完成就触发函数超时，反而增加不一致风险

# 路由注册：每个云函数对应一个 Flask 路由
@app.route('/api/login', methods=['POST'])
def login():
    event = request.get_json()
    # 调用云函数入口逻辑（与 CloudBase 部署时完全相同）
    from functions.login import main
    result = main(event, None)
    return jsonify(result)

@app.route('/api/updateOrderStatus', methods=['POST'])
def update_order_status():
    event = request.get_json()
    from functions.updateOrderStatus import main
    result = main(event, None)
    return jsonify(result)

# ... 其他路由同理

if __name__ == '__main__':
    port = int(os.environ.get('FLASK_PORT', 5000))
    app.run(debug=True, port=port)
```

**设计要点：**

- Flask 路由处理函数仅负责参数转换（`request.get_json()` → `event`），不包含任何业务逻辑
- 业务逻辑完全复用云函数的 `main(event, context)` 函数，`context` 参数在本地模式下传 `None`
- 这保证了本地开发和 CloudBase 部署运行的是同一份代码

### 8.3 db_adapter 双模式切换机制

`db_adapter.py` 是本地/生产双模式的核心，通过工厂模式根据 `DB_MODE` 环境变量选择底层实现：

**接口设计：**

```python
# common/db_adapter.py

class DBAdapterBase:
    """数据库适配器基类，定义统一接口"""

    def get_order(self, order_id: str) -> dict:
        """查询订单"""
        raise NotImplementedError

    def update_order(self, order_id: str, data: dict, condition: dict = None) -> bool:
        """条件更新订单，返回是否成功"""
        raise NotImplementedError

    def push(self, value) -> object:
        """返回一个数组追加操作标记（适配不同数据库的数组操作语法）"""
        raise NotImplementedError

    def query_orders(self, filters: dict, page: int, page_size: int) -> tuple:
        """分页查询订单，返回 (list, total)"""
        raise NotImplementedError


class SQLiteAdapter(DBAdapterBase):
    """SQLite 后端实现，用于本地开发"""

    def __init__(self, db_path: str):
        import sqlite3
        self.conn = sqlite3.connect(db_path)
        self.conn.row_factory = sqlite3.Row

    def get_order(self, order_id: str) -> dict:
        # SQLite 查询实现
        ...

    def update_order(self, order_id: str, data: dict, condition: dict = None) -> bool:
        # SQLite 条件更新实现（通过 WHERE version = ? 实现乐观锁）
        if not self._check_condition(order_id, condition):
            raise VersionConflictError("版本冲突")
        ...


class CloudBaseAdapter(DBAdapterBase):
    """CloudBase 后端实现，用于生产环境"""

    def __init__(self):
        from cloudbase import cloudbase
        self.db = cloudbase.database()

    def get_order(self, order_id: str) -> dict:
        return self.db.collection('orders').document(order_id).get()

    def update_order(self, order_id: str, data: dict, condition: dict = None) -> bool:
        return self.db.collection('orders').document(order_id).update(
            data=data, condition=condition
        )


class VersionConflictError(Exception):
    """乐观锁版本冲突异常"""
    pass


# 工厂函数：根据环境变量自动选择适配器
_adapter = None

def init_db():
    global _adapter
    db_mode = os.environ.get('DB_MODE', 'local')
    if db_mode == 'local':
        db_path = os.environ.get('SQLITE_DB_PATH', 'data/local.db')
        _adapter = SQLiteAdapter(db_path)
    else:
        _adapter = CloudBaseAdapter()

def get_order(order_id: str) -> dict:
    return _adapter.get_order(order_id)

def update_order(order_id: str, data: dict, condition: dict = None) -> bool:
    return _adapter.update_order(order_id, data, condition)

def push(value) -> object:
    return _adapter.push(value)
```

**切换流程：**

```
DB_MODE=local      →  SQLiteAdapter  →  data/local.db
DB_MODE=production →  CloudBaseAdapter  →  CloudBase 云数据库
```
