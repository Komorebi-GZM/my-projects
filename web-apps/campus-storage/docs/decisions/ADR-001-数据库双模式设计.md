# ADR-001: 数据库双模式设计

## Status
Accepted

## Date
2026-05-10

## Context

校园物品暂存平台需要同时支持：
1. **本地开发**：快速迭代、完整测试、离线开发
2. **生产部署**：CloudBase云开发环境，无服务器架构

两种环境的底层数据库技术完全不同：
- 本地开发：SQLite（关系型、文件型、零配置）
- 生产环境：CloudBase NoSQL（文档型、云托管、类MongoDB语法）

## Decision

采用**数据库适配器模式**，通过抽象层屏蔽底层数据库差异，业务代码不直接操作任何具体数据库。

### 架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                      Business Layer                         │
│              (order_service.py, routes/*.py)                │
└─────────────────────────────┬───────────────────────────────┘
                              │
                    ┌─────────▼──────────┐
                    │   DBAdapter (抽象)  │
                    │   - insert()        │
                    │   - update()        │
                    │   - query()         │
                    │   - delete()        │
                    └─────────┬──────────┘
                              │
              ┌───────────────┴───────────────┐
              ▼                               ▼
    ┌─────────────────────┐      ┌──────────────────────┐
    │  SQLiteAdapter      │      │  CloudBaseAdapter    │
    │  - 本地文件存储      │      │  - 云端NoSQL         │
    │  - SQL语法          │      │  - 文档模型          │
    │  - 事务支持         │      │  - 自动扩展          │
    └─────────────────────┘      └──────────────────────┘
```

### 切换机制

通过环境变量 `DB_MODE` 控制：

```python
# factory.py
def get_db_adapter():
    mode = os.getenv("DB_MODE", "local").lower()
    if mode == "production":
        return CloudBaseAdapter(env_id=os.getenv("TCB_ENV_ID"))
    return SQLiteAdapter("data/dev.db")
```

### 数据映射规则

| SQLite | CloudBase NoSQL | 说明 |
|--------|----------------|------|
| `id` (INTEGER PK) | `_id` (String) | 主键转换为字符串 |
| `openid` | `_openid` + `openid` | 保留openid用于查询 |
| `snake_case` 字段 | `camelCase` 字段 | 命名风格转换 |
| `DATETIME` | ISO 8601 String | 时间格式保持兼容 |
| `JSON` 字段 | 嵌套文档 | 直接存储为对象 |

## Alternatives Considered

### 方案A：统一使用CloudBase NoSQL
- **Pros**: 开发环境与生产一致，无适配层复杂度
- **Cons**: 本地开发需要网络连接，测试依赖云服务，CI/CD复杂
- **Rejected**: 开发体验差，离线无法工作

### 方案B：统一使用SQLite
- **Pros**: 简单，本地开发和测试都很方便
- **Cons**: 无法部署到CloudBase，需要额外寻找托管方案
- **Rejected**: 不符合项目使用CloudBase的要求

### 方案C：使用ORM（SQLAlchemy + 云适配）
- **Pros**: ORM提供更高层抽象
- **Cons**: CloudBase NoSQL不完全兼容SQL，ORM支持有限；增加学习成本
- **Rejected**: 过度设计，适配层足够满足需求

## Consequences

### Positive
- 业务代码100%复用，无需为不同环境写两套逻辑
- 本地开发完全离线，测试速度快
- 生产部署无缝切换，只需修改环境变量
- 未来更换数据库只需新增Adapter实现

### Negative
- 需要维护两套Adapter实现
- 复杂查询可能在两种数据库表现不一致
- 事务支持在NoSQL中较弱
- 需要数据迁移脚本同步数据

### Mitigations
- Adapter层充分测试，确保行为一致
- 避免复杂SQL，使用简单CRUD操作
- 重要操作添加补偿机制
- 数据迁移脚本支持--dry-run验证
