# ADR-002: 订单状态机实现

## Status
Accepted

## Date
2026-05-10

## Context

订单是系统的核心实体，具有复杂的状态流转：
- PENDING（待揽收）→ COLLECTED（已揽收）→ STORED（已入库）→ DELIVERING（配送中）→ COMPLETED（已完成）
- 支持取消：PENDING → CANCELLED
- 需要防止非法状态跳转（如：已完成不能回到待揽收）

状态流转涉及业务规则验证：
- 只有管理员可以更新状态
- 某些状态需要前置条件（如配送需要有配送地址）
- 终态（COMPLETED/CANCELLED）不可变更

## Decision

采用**显式状态机模式**，在代码中明确定义状态、转换规则和验证逻辑。

### 状态定义

```python
class OrderStatus:
    PENDING = "PENDING"           # 待揽收
    COLLECTED = "COLLECTED"       # 已揽收
    STORED = "STORED"             # 已入库
    DELIVERING = "DELIVERING"     # 配送中
    COMPLETED = "COMPLETED"       # 已完成（终态）
    CANCELLED = "CANCELLED"       # 已取消（终态）
```

### 转换矩阵

```python
VALID_TRANSITIONS = {
    OrderStatus.PENDING: [OrderStatus.COLLECTED, OrderStatus.CANCELLED],
    OrderStatus.COLLECTED: [OrderStatus.STORED],
    OrderStatus.STORED: [OrderStatus.DELIVERING],
    OrderStatus.DELIVERING: [OrderStatus.COMPLETED],
    OrderStatus.COMPLETED: [],  # 终态
    OrderStatus.CANCELLED: [],  # 终态
}
```

### 验证逻辑

```python
def can_transition(current: str, new: str) -> bool:
    if current in [OrderStatus.COMPLETED, OrderStatus.CANCELLED]:
        return False  # 终态不可变更
    return new in VALID_TRANSITIONS.get(current, [])
```

### 状态历史记录

每次状态变更记录到 `status_history` 字段：

```json
{
  "statusHistory": [
    {"status": "PENDING", "timestamp": "2024-05-10T10:00:00Z", "note": "订单创建"},
    {"status": "COLLECTED", "timestamp": "2024-05-10T14:00:00Z", "note": "快递员已揽收"},
    {"status": "STORED", "timestamp": "2024-05-10T16:00:00Z", "note": "已入库"}
  ]
}
```

## Alternatives Considered

### 方案A：状态字段自由更新
- **Pros**: 实现简单，无限制
- **Cons**: 容易出现非法状态流转，业务规则分散在各处
- **Rejected**: 无法保证数据一致性

### 方案B：数据库触发器/约束
- **Pros**: 数据库层面强制约束
- **Cons**: SQLite和NoSQL支持程度不同，逻辑分散，难以维护
- **Rejected**: 跨数据库兼容性差

### 方案C：工作流引擎（如Camunda）
- **Pros**: 功能强大，支持复杂流程
- **Cons**: 引入重量级依赖，过度设计
- **Rejected**: 项目规模小，状态流转简单

## Consequences

### Positive
- 状态流转规则集中管理，易于理解和维护
- 非法操作在代码层就被拦截，数据一致性有保障
- 状态历史可追溯，便于问题排查
- 单元测试可以覆盖所有状态转换路径

### Negative
- 新增状态或转换规则需要修改代码
- 状态机逻辑与业务代码耦合
- 复杂业务场景可能需要扩展设计

### Mitigations
- 状态机逻辑独立封装在 `order_state.py`
- 使用策略模式预留扩展点
- 文档中维护状态图，变更时同步更新
