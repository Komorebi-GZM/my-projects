## 一、文档概述

### 1.1 定时任务架构

本项目采用**本地Python开发 → 统一部署**模式，定时任务在不同阶段使用不同的实现方案：

- **本地开发阶段**：使用 APScheduler 在 Flask 进程内模拟定时任务，无需依赖云服务即可完成完整的定时任务开发和调试。
- **生产环境**：部署至 CloudBase 后，使用 CloudBase 定时触发器（Cron表达式）驱动，定时任务本质是定时触发的云函数。

两种方案共享同一套业务逻辑（`update_order_status` 等核心函数），仅在调度层做适配，确保本地与生产行为一致。

### 1.2 设计原则

1. **幂等执行**：同一订单多次触发不产生副作用
2. **单条失败不阻塞**：批量处理时单条失败不影响其他订单
3. **限流保护**：避免一次性处理过多订单导致超时
4. **可观测性**：每次执行记录处理数量和失败日志

## 三、任务详细设计

### 3.1 PENDING超时取消

> 引用自《订单状态机设计文档.md》v2.0，第七节自动化规则、决策Q11

**触发规则**：
- 超时条件：当前时间 > 订单创建时间 + 交付窗口期结束时间 + 7天
- 未选交付窗口期：直接以创建时间计7天

**执行逻辑（伪代码描述）**：

```python
from datetime import datetime, timedelta

def main(event=None, context=None):
    # Step 1: 查询超时订单（通过 db_adapter 通用接口，兼容本地SQLite与生产CloudBase）
    timeout_threshold = datetime.utcnow() - timedelta(days=7)
    orders = db_adapter.query_orders(
        status='PENDING',
        create_time_lt=timeout_threshold.isoformat(),
        limit=50
    )

    success_count = 0
    fail_count = 0

    # Step 2: 逐条处理
    for order in orders:
        try:
            if order.get('isPaid'):
                # 已支付：调用update_order_status，触发自动退款逻辑
                update_order_status(
                    order['_id'], 'CANCELLED', 'SYSTEM',
                    reason='TIMEOUT_CANCEL'
                )
            else:
                # 未支付：直接调用update_order_status
                update_order_status(
                    order['_id'], 'CANCELLED', 'SYSTEM',
                    reason='TIMEOUT_CANCEL'
                )
            success_count += 1
            print(f"[pendingTimeoutChecker] 处理成功: {order['_id']}")
        except Exception as error:
            fail_count += 1
            print(f"[pendingTimeoutChecker] 处理失败: {order['_id']}, error: {error}")

    # Step 3: 输出执行摘要
    print(f"[pendingTimeoutChecker] 执行摘要: 总数={len(orders)}, 成功={success_count}, 失败={fail_count}")
```

**状态变更**：PENDING -> CANCELLED

**操作人类型**：SYSTEM

**变更原因**：TIMEOUT_CANCEL

**退款处理**：
- 已支付订单：标记待退款，Demo阶段记录日志即可
- 生产环境：对接微信支付退款API

### 3.2 DELIVERING超时完结

> 引用自《订单状态机设计文档.md》v2.0，决策Q4

**触发规则**：
- 超时条件：当前时间 > statusHistory中DELIVERING状态的timestamp + 7天
- 计时起点：以statusHistory数组中toStatus === 'DELIVERING'的记录的timestamp为准

**执行逻辑（伪代码描述）**：

```python
from datetime import datetime, timedelta

def main(event=None, context=None):
    # Step 1: 查询超时订单（通过 db_adapter 通用接口，兼容本地SQLite与生产CloudBase）
    timeout_threshold = datetime.utcnow() - timedelta(days=7)
    orders = db_adapter.query_orders(
        status='DELIVERING',
        limit=50
    )

    # 在内存中过滤：statusHistory中DELIVERING记录的timestamp < (now - 7天)
    filtered_orders = []
    for order in orders:
        for record in order.get('statusHistory', []):
            if record.get('toStatus') == 'DELIVERING':
                record_time = datetime.fromisoformat(record['timestamp'].replace('Z', '+00:00'))
                if record_time < timeout_threshold:
                    filtered_orders.append(order)
                break

    success_count = 0
    fail_count = 0

    # Step 2: 逐条处理
    for order in filtered_orders:
        try:
            update_order_status(
                order['_id'], 'COMPLETED', 'SYSTEM',
                reason='AUTO_COMPLETE'
            )
            success_count += 1
            print(f"[deliveringTimeoutChecker] 处理成功: {order['_id']}")
        except Exception as error:
            fail_count += 1
            print(f"[deliveringTimeoutChecker] 处理失败: {order['_id']}, error: {error}")

    # Step 3: 输出执行摘要
    print(f"[deliveringTimeoutChecker] 执行摘要: 总数={len(filtered_orders)}, 成功={success_count}, 失败={fail_count}")
```

**状态变更**：DELIVERING -> COMPLETED

**操作人类型**：SYSTEM

**变更原因**：AUTO_COMPLETE

## 四、实现方案

### 4.1 本地开发方案：APScheduler + Flask 集成

本地开发阶段使用 APScheduler 在 Flask 应用进程内运行定时任务，无需外部依赖。

```python
# scheduler_config.py
from apscheduler.schedulers.background import BackgroundScheduler
from apscheduler.triggers.interval import IntervalTrigger
from tasks.timeout_checker import check_pending_timeout, check_delivering_timeout

def init_scheduler(app):
    """在 Flask 应用启动时初始化定时任务调度器"""
    scheduler = BackgroundScheduler()

    # 每30分钟执行一次 PENDING 超时取消检查
    scheduler.add_job(
        check_pending_timeout,
        trigger=IntervalTrigger(minutes=30),
        id='pending_timeout_checker',
        replace_existing=True
    )

    # 每30分钟执行一次 DELIVERING 超时完结检查
    scheduler.add_job(
        check_delivering_timeout,
        trigger=IntervalTrigger(minutes=30),
        id='delivering_timeout_checker',
        replace_existing=True
    )

    scheduler.start()
    app.scheduler = scheduler  # 挂载到 app 以便需要时手动触发
```

```python
# app.py
from flask import Flask
from scheduler_config import init_scheduler

app = Flask(__name__)
init_scheduler(app)

@app.route('/')
def index():
    return 'App is running'
```

**优势**：
- 与 Flask 进程共享数据库连接和业务逻辑，无需 HTTP 调用
- 支持在 IDE 中断点调试定时任务逻辑
- 可随时调整触发频率，方便开发测试

### 4.2 生产环境方案：CloudBase 定时触发器

| 任务 | 云函数名 | Cron表达式 | 说明 |
|------|---------|-----------|------|
| PENDING超时取消 | `pendingTimeoutChecker` | `0 */30 * * * * *` | 每30分钟执行 |
| DELIVERING超时完结 | `deliveringTimeoutChecker` | `0 */30 * * * * *` | 每30分钟执行 |

CloudBase定时触发器使用7位Cron表达式（含秒）。

### 4.3 并发控制

- 本地开发：APScheduler 默认单线程执行，无并发问题
- 生产环境：CloudBase 定时触发器单实例执行（CloudBase默认行为）
- 每次处理限制50条，防止超时
- 处理时间预估：50条 x 200ms/条 = 10秒（远低于20秒超时）

### 4.4 冷启动说明

- **本地开发环境**：Flask 进程常驻运行，APScheduler 任务无冷启动问题，首次执行即可立即响应。
- **生产环境（CloudBase）**：Python 云函数存在冷启动现象，首次调用或长时间未调用后，函数实例需要重新初始化：
  - **冷启动耗时**：约300~800ms（取决于依赖包大小和初始化逻辑）
  - **影响范围**：仅影响首次触发或实例回收后的首次调用，后续调用走热实例，无额外延迟
  - **优化建议**：
    - 控制 `requirements.txt` 中的依赖数量，仅安装必要包
    - 将数据库连接、SDK初始化等操作放在模块级别（而非函数内部），利用实例复用
    - 避免在 `main(event, context)` 函数内部执行重量级初始化操作

### 4.5 失败重试

- 单条失败不重试（下次定时任务会重新扫描）
- 记录失败日志便于排查
- 连续多次失败可人工介入

### 4.6 幂等性保障

- 通过 `update_order_status` 函数的幂等性保障
- 状态已变更的订单不会被重复处理（状态校验会拒绝）

### 4.7 本地调试方法

在本地开发阶段，可以通过以下方式手动触发定时任务进行调试：

```python
# 手动触发方式1：直接调用任务函数
from tasks.timeout_checker import check_pending_timeout, check_delivering_timeout

check_pending_timeout()      # 手动触发 PENDING 超时取消
check_delivering_timeout()   # 手动触发 DELIVERING 超时完结

# 手动触发方式2：通过 Flask 管理命令
# flask run 时在另一个终端执行：
# python -c "from tasks.timeout_checker import check_pending_timeout; check_pending_timeout()"
```

也可在 Flask 应用中注册调试路由（仅开发环境启用）：

```python
# 仅在开发环境注册调试路由
if app.config.get('ENV') == 'development':
    @app.route('/debug/trigger/<task_name>', methods=['POST'])
    def debug_trigger_task(task_name):
        from tasks.timeout_checker import TASK_MAP
        if task_name in TASK_MAP:
            TASK_MAP[task_name]()
            return {'status': 'ok', 'task': task_name}
        return {'status': 'error', 'message': 'unknown task'}, 404
```

## 六、与update_order_status的关系

定时任务本质是SYSTEM角色调用 `update_order_status` 函数，复用其状态转换校验、乐观锁、日志记录等所有逻辑。

- **本地开发**：直接以函数调用方式执行 `update_order_status(orderId, newStatus, "SYSTEM", "system", reason)`，不经过 HTTP，与 Flask 进程共享内存和数据库连接，调试方便。
- **生产环境**：定时触发器触发云函数入口 `main(event, context)`，在云函数内部同样以函数调用方式执行 `update_order_status`。

```
# 本地开发
APScheduler -> check_pending_timeout() -> update_order_status(orderId, newStatus, "SYSTEM", "system", reason)

# 生产环境
CloudBase定时触发器 -> main(event, context) -> update_order_status(orderId, newStatus, "SYSTEM", "system", reason)
```

两种模式下，`update_order_status` 的调用方式完全一致，差异仅在于调度入口。
