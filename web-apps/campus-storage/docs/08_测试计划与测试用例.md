## 一、文档概述

### 1.1 测试范围

- 本地单元测试（`updateOrderStatus` 状态机逻辑为核心）
- 本地联调测试（Flask + 微信开发者工具）
- 生产环境部署后验证

### 1.2 测试策略

采用"本地Python开发"模式，以本地自动化测试为主：

1. 状态机所有合法转换（正向测试）
2. 状态机所有非法转换（反向测试）
3. 权限校验（越权操作测试）
4. 并发场景（乐观锁测试）
5. 超时自动处理（定时任务测试）

### 1.3 测试环境

| 环境 | 说明 |
|------|------|
| 本地开发环境 | Python 3.12+, pytest, SQLite 内存数据库（`:memory:`），用于单元测试与集成测试 |
| 联调环境 | Flask 本地服务器（`flask run`）+ 微信开发者工具，用于前后端联调 |
| 生产环境 | CloudBase 云开发（部署后验证），用于最终生产验收 |

## 四、状态机测试用例（⭐重点）

> 以下用例基于《订单状态机设计文档 v2.0》中的状态定义、转换规则与权限控制设计。
> 状态机共8种状态：`PENDING`、`COLLECTED`、`TRANSIT`、`STORED`、`DELIVERING`、`COMPLETED`（终态）、`CANCELLED`（终态）、`EXCEPTION`。
> 终态 `COMPLETED` 与 `CANCELLED` 不可逆；`EXCEPTION` 仅管理员可操作，且必须填写原因。

### 4.1 正向流程测试

| 用例ID | 用例名称 | 测试路径 | 预期结果 |
|--------|---------|---------|---------|
| TC-SM-001 | 完整正向流程 | `PENDING`→`COLLECTED`→`TRANSIT`→`STORED`→`DELIVERING`→`COMPLETED` | 每步状态正确变更，`statusHistory` 记录完整，各时间字段（`collectedAt`、`transitAt`、`storedAt`、`deliveringAt`、`completedAt`）正确写入 |
| TC-SM-002 | 用户取消流程（已支付） | `PENDING`(isPaid=true)→`CANCELLED` | 取消成功，`cancelledAt` 记录时间，触发自动退款，`cancelReason` 为 `USER_CANCEL` |
| TC-SM-003 | 用户取消流程（未支付） | `PENDING`(isPaid=false)→`CANCELLED` | 取消成功，无退款逻辑，`cancelReason` 为 `USER_CANCEL` |
| TC-SM-004 | 异常结案流程 | `PENDING`→`EXCEPTION`→`COMPLETED` | 异常标记成功（管理员操作，`reason` 必填），结案成功，`statusHistory` 包含两条变更记录 |
| TC-SM-005 | 异常作废流程 | `PENDING`→`EXCEPTION`→`CANCELLED` | 异常标记成功，作废成功，已支付订单触发退款 |
| TC-SM-006 | 各状态均可标记异常 | 分别从 `COLLECTED`/`TRANSIT`/`STORED`/`DELIVERING`→`EXCEPTION` | 仅管理员可操作，标记成功，`reason` 必填，操作日志完整 |

### 4.4 并发测试

> 根据状态机设计决策Q7：使用乐观锁机制，本地开发使用 SQLite 行版本号实现，重试最多3次。

| 用例ID | 用例名称 | 测试场景 | 预期结果 |
|--------|---------|---------|---------|
| TC-CONC-001 | 同一订单并发状态变更 | 2个请求同时将 `PENDING`→`COLLECTED` | 一个成功，一个返回错误码2003（版本冲突） |
| TC-CONC-002 | 乐观锁重试成功 | 模拟版本号冲突后重试（≤3次） | 重试成功，`statusHistory` 无重复记录 |
| TC-CONC-003 | 乐观锁重试耗尽 | 模拟版本号连续冲突3次 | 返回错误码2003，状态不变 |
| TC-CONC-004 | 幂等性验证 | 相同请求连续发送2次 | 第2次返回成功但不重复写入 `statusHistory` |
| TC-CONC-005 | 不同状态并发变更 | 请求A：`PENDING`→`COLLECTED`，请求B：`PENDING`→`CANCELLED` | 一个成功，另一个返回2003 |

## 五、定时任务测试用例

> 根据状态机设计：自动化规则包括发运提醒（`COLLECTED` 超24h）、自动完结（`DELIVERING` 超7天）、超时取消（`PENDING` 超交付窗口期7天）。

| 用例ID | 用例名称 | 测试场景 | 预期结果 |
|--------|---------|---------|---------|
| TC-TIMER-001 | `PENDING` 超时取消（未支付） | 创建7天前的 `PENDING`(isPaid=false) 订单→触发定时任务 | 订单自动转 `CANCELLED`，无退款逻辑 |
| TC-TIMER-002 | `PENDING` 超时取消（已支付） | 创建7天前的 `PENDING`(isPaid=true) 订单→触发定时任务 | 订单自动转 `CANCELLED`，记录退款日志，`cancelReason` 为 `TIMEOUT_CANCEL` |
| TC-TIMER-003 | `PENDING` 未超时不取消 | 创建1天前的 `PENDING` 订单→触发定时任务 | 订单状态不变 |
| TC-TIMER-004 | `DELIVERING` 超时自动完结 | 创建7天前的 `DELIVERING` 订单→触发定时任务 | 订单自动转 `COMPLETED`，以 `statusHistory` 中 `DELIVERING` 的 `timestamp` 为起点计算 |
| TC-TIMER-005 | `DELIVERING` 未超时不完结 | 创建3天前的 `DELIVERING` 订单→触发定时任务 | 订单状态不变 |
| TC-TIMER-006 | 定时任务幂等性（已取消） | 已 `CANCELLED` 的订单再次被扫描 | 不重复处理，状态不变 |
| TC-TIMER-007 | 定时任务幂等性（已完成） | 已 `COMPLETED` 的订单再次被扫描 | 不重复处理，状态不变 |
| TC-TIMER-008 | `COLLECTED` 超24h发运提醒 | 创建25小时前的 `COLLECTED` 订单→触发定时任务 | 系统发送提醒通知给运营人员 |
| TC-TIMER-009 | `COLLECTED` 未超24h不发提醒 | 创建12小时前的 `COLLECTED` 订单→触发定时任务 | 不发送提醒 |
| TC-TIMER-010 | 交付窗口期计时验证 | 创建订单时选择3天后的交付窗口期→距窗口期结束不足7天→触发定时任务 | 订单状态不变（以创建时间+窗口期结束时间为计时起点） |

## 九、操作日志（statusHistory）验证要点

> 每次状态变更均应写入 `statusHistory` 数组，测试时需验证以下字段完整性：

| 验证字段 | 说明 | 验证方法 |
|---------|------|---------|
| `timestamp` | 变更时间 | 与系统时间对比，误差 < 5s |
| `fromStatus` | 原状态 | 与变更前订单 `status` 一致 |
| `toStatus` | 新状态 | 与变更后订单 `status` 一致 |
| `operatorType` | 操作人类型 | 枚举值：`USER` / `ADMIN` / `SYSTEM` |
| `operatorId` | 操作人ID | 与当前登录用户/管理员ID一致 |
| `reason` | 变更原因 | 枚举约束：`USER_CANCEL`、`TIMEOUT_CANCEL`、`ADMIN_DISPATCH`、`DELIVERY_FAILED`、`ITEM_DAMAGE`、`ADMIN_MARK_EXCEPTION` 等 |
| `metadata` | 附加信息 | 如物流单号、仓库位置、入库照片URL等 |

| 错误码 | 含义 | 触发场景 |
|--------|------|---------|
| 1003 | 权限不足 | 用户尝试执行管理员专属操作 |

| 2004 | 终态不可变更 | 尝试修改 `COMPLETED` 或 `CANCELLED` 状态的订单 |

- [ ] 退款流程是否在本Demo阶段实现真实微信支付退款（或仅记录退款日志）
- [ ] SQLite 种子脚本是否需要覆盖更多边界场景数据

## 十四、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| SQLite 与 CloudBase 行为差异 | 本地测试通过但生产环境出现兼容性问题 | 核心逻辑使用抽象数据访问层，生产部署前增加 CloudBase 集成测试；重点关注事务隔离级别、查询语法差异（如数组查询、分页方式） |
| 乐观锁在高并发下重试耗尽 | 状态变更失败 | 重试3次后返回明确错误，前端引导用户重试 |
| 定时任务触发延迟 | 超时订单未及时处理 | 本地使用 APScheduler 调试，生产环境部署后监控执行日志，设置告警 |
| `statusHistory` 数组无限增长 | 数据库性能下降 | Demo阶段可接受，生产环境需考虑归档策略 |
| 微信登录态过期 | 测试中断 | 联调测试前确认登录态有效，必要时重新登录 |
