## 一、状态机流转图（Mermaid）

```mermaid
stateDiagram-v2
    direction LR
    [*] --> PENDING : 创建订单（支付由isPaid字段管理）

    %% 正向主流程
    PENDING --> COLLECTED : 运营确认取件/用户自送
    COLLECTED --> TRANSIT : 运营手动发运（24h未操作提醒）
    TRANSIT --> STORED : 到仓拍照入库
    STORED --> DELIVERING : 用户发起配送
    DELIVERING --> COMPLETED : 手动确认 / 7天超时自动完结（以DELIVERING创建时间计）

    %% 取消流程（仅PENDING可主动取消）
    PENDING --> CANCELLED : 主动取消 / 7天超时（计时：订单创建+交付窗口期结束）
    note over PENDING,CANCELLED : 未支付直接取消；已支付取消并自动退款

    %% 异常：所有状态可标记（管理员专属）
    PENDING --> EXCEPTION : 管理员标记异常
    COLLECTED --> EXCEPTION : 管理员标记异常
    TRANSIT --> EXCEPTION : 管理员标记异常
    STORED --> EXCEPTION : 管理员标记异常
    DELIVERING --> EXCEPTION : 管理员标记异常（含配送失败退回）

    %% 异常闭环（仅管理员操作）
    EXCEPTION --> COMPLETED : 人工结案（如：破损赔偿后完结）
    EXCEPTION --> CANCELLED : 人工作废（如：配送失败→取消+退款+重下单）

    %% 终态约束
    COMPLETED : 【终态】不可逆｜投诉走独立工单
    CANCELLED : 【终态】不可逆
    EXCEPTION : 【仅管理员】必填原因+操作日志
```

## 二、状态定义

| 状态码 | 状态名称 | 说明 | 可转换目标 |
|--------|---------|------|-----------|
| `PENDING` | 待交付 | 订单已创建，等待取件或用户自送 | `COLLECTED`, `CANCELLED`, `EXCEPTION` |
| `COLLECTED` | 已揽收 | 运营已确认取件，等待发运 | `TRANSIT`, `EXCEPTION` |
| `TRANSIT` | 运输中 | 物品已发运，正在运往仓库 | `STORED`, `EXCEPTION` |
| `STORED` | 已入库 | 物品已到仓并拍照入库 | `DELIVERING`, `EXCEPTION` |
| `DELIVERING` | 配送中 | 用户已发起配送，物品正在配送途中 | `COMPLETED`, `EXCEPTION` |
| `COMPLETED` | 已完成 | **【终态】** 订单正常完结 | 无（终态） |
| `CANCELLED` | 已取消 | **【终态】** 订单已取消 | 无（终态） |
| `EXCEPTION` | 异常状态 | **【仅管理员】** 需人工介入处理 | `COMPLETED`, `CANCELLED` |

## 三、状态转换规则

### 3.1 正向主流程

```
创建订单 → PENDING → COLLECTED → TRANSIT → STORED → DELIVERING → COMPLETED
```

| 转换 | 触发条件 | 操作说明 |
|------|---------|---------|
| `[*] → PENDING` | 用户创建订单 | 支付状态由 `isPaid` 字段独立管理 |
| `PENDING → COLLECTED` | 运营确认取件 / 用户自送到仓 | 更新取件时间、取件人信息 |
| `COLLECTED → TRANSIT` | 运营手动发运 | **24h未操作需系统提醒** |
| `TRANSIT → STORED` | 到仓拍照入库 | 上传入库照片，记录仓库位置 |
| `STORED → DELIVERING` | 用户发起配送 | 用户填写配送地址并支付配送费 |
| `DELIVERING → COMPLETED` | 手动确认 / 7天超时 | 以 `DELIVERING` 状态创建时间计 |

### 3.2 取消流程（仅PENDING可主动取消）

| 转换 | 触发条件 | 操作说明 |
|------|---------|---------|
| `PENDING → CANCELLED` | 用户主动取消 / 7天超时 | 计时：订单创建时间 + 交付窗口期结束 |

**取消退款规则：**
- 未支付订单：直接取消，无需退款
- 已支付订单：取消并自动退款

> **注意**：`COLLECTED` 及之后状态不可由用户直接取消。用户需发起取消申请 → 管理员转 `EXCEPTION` → 最终 `CANCELLED`。

### 3.3 异常流程

**触发条件：** 管理员标记异常（所有非终态均可）

**操作要求：**
- 必填异常原因（枚举约束，见决策Q10）
- 记录完整操作日志（含 `operatorType`, `operatorId`, `reason`, `metadata`）
- 云函数鉴权校验 `operatorType === 'ADMIN'`

**异常闭环：**

| 转换 | 场景示例 |
|------|---------|
| `EXCEPTION → COMPLETED` | 破损赔偿后完结、丢失补发后确认收货 |
| `EXCEPTION → CANCELLED` | 配送失败无法恢复 → 取消 + 退款 + 建议重新下单 |

> **核心原则**：异常不死亡，均有闭环。

## 五、15项核心设计决策

| 编号 | 问题 | 决策 | 理由/边界处理 |
|------|------|------|-------------|
| Q1 | PENDING是否独立于支付状态？ | **是**，支付用`isPaid`字段管理，与状态机解耦 | 未支付PENDING同样受超时取消约束，取消时区分是否退款 |
| Q2 | COLLECTED是否拆分"已取件+运输中"？ | **是**，拆为`COLLECTED`（已取件）和`TRANSIT`（运输中） | 解决物流灰色地带，操作者明确 |
| Q3 | COLLECTED状态是否可以取消？ | **否**，用户需发起取消申请→人工转`EXCEPTION`→最终`CANCELLED` | 避免自动化处理已交付物品取消的复杂性 |
| Q4 | COMPLETED触发方式？ | **手动确认 + 7天超时自动完结** | 超时计时起点：`DELIVERING`状态创建时间 |
| Q5 | COMPLETED是否可转EXCEPTION？ | **否**，收货后投诉走独立工单系统，不修改订单状态 | 状态机与投诉工单解耦 |
| Q6 | EXCEPTION出口路径？ | **人工结案→COMPLETED** 或 **人工作废→CANCELLED** | 异常不死亡，均有闭环 |
| Q7 | 乐观锁实现？ | CloudBase `_version`乐观锁，云函数内重试最多3次 | 保证并发状态变更一致性 |
| Q8 | ABA问题评估？ | 状态无逆向回流，当前设计**无ABA风险** | 已评估并记录 |
| Q9 | 操作人类型？ | `USER` / `ADMIN` / `SYSTEM` 三选一 | 明确每个状态变更的触发主体 |
| Q10 | 状态变更原因枚举？ | 枚举约束：`USER_CANCEL`、`TIMEOUT_CANCEL`、`ADMIN_DISPATCH`、`DELIVERY_FAILED`、`ITEM_DAMAGE`、`ADMIN_MARK_EXCEPTION` 等 | 强制传入，日志完整可追溯 |
| Q11 | PENDING超时时长与计时起点？ | **7天**，计时起点 = `订单创建时间 + 用户选择的交付窗口期结束时间` | 未选窗口期则直接以创建时间计 |
| Q12 | 配送失败兜底？ | 管理员标记`EXCEPTION`（原因：`DELIVERY_FAILED`）→ 人工取消→用户重新下单 | 不走逆向状态流转 |
| Q13 | 多物品部分入库？ | **不支持**，1订单 = 1包裹，多包裹拆分下单 | 明确粒度，避免状态机复杂化 |
| Q14 | 后台状态→用户文案映射？ | 已做映射表，`EXCEPTION`对外统一显示「处理中」 | 不暴露内部异常细节 |
| Q15 | 异常操作权限？ | 仅管理员可标记异常、结案、作废；云函数鉴权；`reason`必填 | 权限校验+操作日志 |

## 六、终态约束

| 终态 | 特性 | 后续处理 |
|------|------|---------|
| `COMPLETED` | 不可逆 | 投诉走独立工单系统，不改订单状态 |
| `CANCELLED` | 不可逆 | 无 |

---

## 七、自动化规则

| 规则 | 触发条件 | 执行动作 |
|------|---------|---------|
| 发运提醒 | `COLLECTED` 状态超过24小时 | 系统提醒运营人员 |
| 自动完结 | `DELIVERING` 状态超过7天 | 自动转为 `COMPLETED`（以 `statusHistory` 中 `DELIVERING` 的 `timestamp` 为起点） |
| 超时取消 | `PENDING` 状态超过交付窗口期（7天） | 自动转为 `CANCELLED`，区分 `isPaid` 决定是否自动退款 |

|--------|------|------|
| `status` | Enum | 订单状态（上述8种） |

## 十一、编码注意事项

1. **乐观锁重试的幂等性**：重试前检查 `statusHistory` 末条记录，避免同一次变更写入多条日志
2. **管理员鉴权硬编码**：`EXCEPTION → COMPLETED/CANCELLED` 必须在云函数中校验 `operatorType === 'ADMIN'`
3. **COMPLETED状态保留投诉入口**：用户端展示"投诉/报损"按钮，调用独立工单接口，不改订单状态
4. **PENDING超时取消逻辑**：定时触发器检查超时，区分 `isPaid` 决定是否自动退款
5. **DELIVERING超时完结**：以 `statusHistory` 中 `DELIVERING` 状态的 `timestamp` 为起点计算7天

## 十三、状态机设计原则

1. **严谨性优先**：状态转换严格定义，非法转换在代码层面拒绝
2. **异常闭环完整**：异常不死亡，均有 `COMPLETED` 或 `CANCELLED` 出口
3. **状态机与支付解耦**：支付由 `isPaid` 独立管理，不阻塞状态流转
4. **状态机与工单解耦**：投诉走独立工单系统，不修改订单状态
5. **权限分明**：异常操作仅限管理员，操作日志完整可追溯
6. **自动兜底**：超时自动处理，减少人工干预
