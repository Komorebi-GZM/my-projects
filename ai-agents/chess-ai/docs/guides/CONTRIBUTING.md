# 贡献指南与代码规范

## 文档元信息
- **文档版本**：v1.0
- **适用对象**：开源贡献者、开发者、代码提交者
- **项目类型**：Python 开源中国象棋AI桌面应用
- **用途**：规范协作流程、代码标准、提交与审核规则

## 2. 欢迎贡献

### 2.1 项目协作理念

本项目欢迎并感谢所有形式的贡献。无论你是资深开发者还是初次参与开源，都能找到适合自己的贡献方式。

### 2.2 贡献形式

| 贡献类型 | 说明 | 适合人群 |
|---------|------|---------|
| **Bug反馈** | 提交问题报告，帮助改进稳定性 | 所有用户 |
| **功能建议** | 提出新功能需求或改进方案 | 所有用户 |
| **文档优化** | 改进文档、翻译、补充说明 | 所有用户 |
| **代码提交** | 修复Bug、实现功能、重构优化 | 开发者 |
| **代码审核** | 审查PR、提供技术建议 | 资深开发者 |

### 2.3 协作原则

| 原则 | 说明 |
|-----|------|
| **开放** | 接纳所有建设性意见，不设门槛 |
| **友善** | 尊重每位贡献者，保持耐心和礼貌 |
| **高效** | 遵循规范流程，减少沟通成本 |
| **尊重** | 尊重不同技术水平和观点差异 |

## 3. 行为准则

### 3.1 社区公约

本项目遵循 [Contributor Covenant](https://www.contributor-covenant.org/) 贡献者公约。

### 3.2 沟通礼仪

| 鼓励行为 | 禁止行为 |
|---------|---------|
| 建设性技术讨论 | 人身攻击或侮辱性语言 |
| 耐心解答新手问题 | 嘲讽或贬低他人贡献 |
| 坦诚表达不同意见 | 基于性别、种族、背景的歧视 |
| 积极参与代码审核 | 恶意差评或拒绝合理建议 |

### 3.3 包容原则

- 使用性别中立的语言
- 尊重不同的技术背景和经验水平
- 对新手保持耐心，提供引导而非批评
- 遇到违反行为准则的情况，请联系维护者处理

## 4. 贡献方式

### 4.1 Bug反馈

提交Bug Issue时，请包含以下信息：

| 必填项 | 说明 |
|-------|------|
| **问题描述** | 清晰描述Bug现象 |
| **复现步骤** | 一步步说明如何复现 |
| **预期行为** | 正确情况下应该怎样 |
| **实际行为** | 实际发生了什么 |
| **环境信息** | 操作系统、Python版本、AI模型 |

| 可选项 | 说明 |
|-------|------|
| 截图/录屏 | 异常界面截图 |
| 日志文件 | `logs/` 目录下的相关日志 |
| 复现代码 | 最小化复现示例 |

**Issue标签**：提交后维护者会添加标签（`bug` / `P0` / `P1` 等）。

### 4.2 功能提议

提交功能建议Issue时，请说明：

| 必填项 | 说明 |
|-------|------|
| **需求描述** | 清晰描述期望的功能 |
| **使用场景** | 该功能解决什么问题 |
| **价值说明** | 对项目/用户的收益 |

**提议流程**：

```
提交Issue → 社区讨论 → 维护者评估 → 纳入规划（或关闭并说明原因）
```

### 4.3 文档优化

| 优化类型 | 方式 |
|---------|------|
| **错别字/排版** | 直接提交PR |
| **内容补充** | 先提Issue讨论，确认后提交PR |
| **新文档** | 先提Issue说明需求，讨论后编写 |

### 4.4 代码贡献完整流程

```
步骤1: Fork项目仓库
    │
    ▼
步骤2: 创建功能分支（从main分支）
    │
    ▼
步骤3: 开发与本地测试
    │
    ▼
步骤4: 提交代码（遵循提交规范）
    │
    ▼
步骤5: 推送到Fork仓库
    │
    ▼
步骤6: 创建Pull Request
    │
```

## 5. 代码规范

### 5.1 Python编码风格

| 规范项 | 标准 | 工具 |
|-------|------|------|
| **代码风格** | PEP 8 | Ruff / Black |
| **行长度** | 最大120字符 | Ruff |
| **导入排序** | isort标准 | isort |
| **自动格式化** | 提交前自动执行 | pre-commit钩子 |

### 5.2 强制类型注解

| 要求 | 说明 |
|-----|------|
| **函数参数** | 所有参数必须标注类型 |
| **函数返回值** | 所有函数必须标注返回类型 |
| **类属性** | 公开属性使用类型注解 |
| **禁止Any** | 除非有充分理由，禁止使用`Any`类型 |

```python
# ✅ 正确
def get_legal_moves(board: Board, side: Literal["red", "black"]) -> list[Move]:
    ...

# ❌ 错误
def get_legal_moves(board, side):
    ...
```

### 5.3 命名约定

| 类型 | 约定 | 示例 |
|-----|------|------|
| **类名** | PascalCase | `ChessAgent`, `MoveResult` |
| **函数名** | snake_case | `get_legal_moves`, `apply_move` |
| **变量名** | snake_case | `current_fen`, `move_number` |
| **常量** | UPPER_SNAKE_CASE | `MAX_RETRIES`, `DEFAULT_TEMPERATURE` |
| **私有方法** | 单下划线前缀 | `_clean_noise`, `_extract_from_json` |
| **私有属性** | 单下划线前缀 | `_conn`, `_instance` |
| **模块名** | snake_case | `game_controller.py`, `output_parser.py` |

### 5.4 文档字符串规范

所有公开类和方法必须包含文档字符串：

```python
def is_valid_move(board: Board, move: Move) -> bool:
    """
    校验走子是否合法。

    Args:
        board: 当前棋盘状态
        move: 待校验的走子

    Returns:
        合法返回True，非法返回False

    Raises:
        InvalidPositionError: 坐标超出棋盘范围
    """
```

| 要求 | 说明 |
|-----|------|
| **格式** | Google风格docstring |
| **内容** | 简要描述 + Args + Returns + Raises |
| **语言** | 中文描述，参数名保持英文 |
| **私有方法** | 可选，复杂逻辑建议添加 |

### 5.5 项目目录结构与模块职责

| 目录 | 职责 | 规则 |
|-----|------|------|
| `src/core/` | 棋盘规则引擎 | 纯计算，无外部依赖 |
| `src/ai/` | AI代理与LLM集成 | 依赖LangChain/LangGraph |
| `src/gui/` | GUI界面 | 依赖Pygame |
| `src/db/` | 数据持久化 | 依赖SQLite |
| `src/common/` | 公共工具与模型 | 无业务依赖 |
| `tests/` | 测试代码 | 镜像src/结构 |

**依赖规则**：单向依赖，禁止循环引用，禁止跨层直接调用。

## 6. 提交信息规范

### 6.1 标准化格式

```
<类型>(<范围>): <简要描述>

<详细说明（可选）>

<关联Issue（可选）>
```

### 6.2 提交类型定义

| 类型 | 说明 | 示例 |
|-----|------|------|
| `feat` | 新功能 | `feat(ai): 添加流式思考输出` |
| `fix` | 修复Bug | `fix(parser): 修复JSON解析括号不匹配` |
| `docs` | 文档变更 | `docs: 更新贡献指南` |
| `style` | 代码格式（不影响逻辑） | `style: 统一导入排序` |
| `refactor` | 重构（不新增功能不修复Bug） | `refactor(db): 优化事务控制` |
| `test` | 测试相关 | `test(core): 补充困毙判定测试` |
| `chore` | 构建/工具变更 | `chore: 更新依赖版本` |
| `perf` | 性能优化 | `perf(parser): 优化正则匹配性能` |

### 6.3 范围定义

| 范围 | 说明 |
|-----|------|
| `core` | 棋盘规则引擎 |
| `ai` | AI代理与LLM模块 |
| `gui` | GUI界面 |
| `db` | 数据持久化 |
| `parser` | 输出解析器 |
| `config` | 配置管理 |
| `docs` | 文档 |

### 6.4 提交要求

| 要求 | 说明 |
|-----|------|
| **原子性** | 每次提交只做一件事 |
| **简洁性** | 标题不超过72字符 |
| **可读性** | 使用祈使语气（"添加"而非"添加了"） |
| **完整性** | 不提交半成品代码 |
| **关联性** | 关联Issue时在末尾标注 `Closes #XX` |

## 7. Pull Request 流程与规范

### 7.1 PR创建要求

| 必填项 | 说明 |
|-------|------|
| **标题** | 遵循提交信息格式（类型+范围+描述） |
| **描述** | 说明变更内容、动机、测试方式 |
| **关联Issue** | 标注 `Closes #XX` 或 `Related to #XX` |
| **变更类型** | 勾选对应的变更类型标签 |

### 7.2 PR描述模板

```markdown
## 变更说明
<!-- 描述本次PR做了什么 -->

## 动机
<!-- 为什么需要这个变更 -->

## 测试方式
<!-- 如何验证变更正确性 -->

## 关联Issue
Closes #XX

## 检查清单
- [ ] 代码遵循项目规范
- [ ] 已添加/更新测试
- [ ] 所有测试通过
- [ ] 文档已更新（如需要）
```

### 7.3 CI自动化检查

PR提交后自动触发以下检查：

| 检查项 | 要求 | 失败处理 |
|-------|------|---------|
| **代码格式** | Ruff + Black检查通过 | 自动格式化后重新提交 |
| **类型检查** | mypy检查无错误 | 修复类型注解 |
| **单元测试** | 全部通过 | 修复失败用例 |
| **覆盖率** | ≥85% | 补充测试用例 |
| **集成测试** | 全部通过 | 修复接口问题 |

### 7.4 审核与合并

| 审核要点 | 说明 |
|---------|------|
| **代码质量** | 规范性、可读性、可维护性 |
| **逻辑正确** | 功能实现是否符合预期 |
| **测试覆盖** | 新增代码是否有对应测试 |
| **文档更新** | 公开接口是否有文档更新 |

**合并规则**：
- 至少1位维护者审核通过
- 所有CI检查通过
- 无未解决的讨论意见
- Squash合并到main分支

### 7.5 分支管理

| 分支类型 | 命名规范 | 说明 |
|---------|---------|------|
| `main` | - | 主分支，始终保持可发布状态 |
| `feature/*` | `feature/ai-stream-output` | 新功能开发 |
| `fix/*` | `fix/parser-json-error` | Bug修复 |
| `docs/*` | `docs/update-contributing` | 文档更新 |

PR合并后自动删除功能分支。

## 8. 自动化测试要求

### 8.1 测试覆盖规则

| 变更类型 | 测试要求 |
|---------|---------|
| **新功能** | 必须包含单元测试 + 集成测试 |
| **Bug修复** | 必须包含回归测试（复现Bug的测试用例） |
| **重构** | 现有测试必须全部通过 |
| **文档** | 无测试要求 |

### 8.2 覆盖率要求

| 指标 | 最低要求 | 推荐值 |
|-----|---------|-------|
| 行覆盖率 | ≥85% | ≥90% |
| 分支覆盖率 | ≥75% | ≥80% |

### 8.3 测试执行

| 场景 | 命令 | 说明 |
|-----|------|------|
| 全部测试 | `pytest tests/` | 运行完整测试套件 |
| 单元测试 | `pytest tests/unit/` | 仅单元测试 |
| 集成测试 | `pytest tests/integration/` | 仅集成测试 |
| 带覆盖率 | `pytest --cov=src` | 生成覆盖率报告 |
| 指定模块 | `pytest tests/unit/test_board.py` | 运行单个测试文件 |

### 8.4 回归测试保障

- 每次PR合并后自动运行完整测试套件
- 发现回归问题优先处理，必要时回滚合并
- 关键模块（规则引擎、解析器）变更需额外人工验证
